Git Best Practices

Commit Related Changes

A commit should be a wrapper for related changes.

$ git add -p

-p, --patch Use the interactive patch selection interface to chose which changes to commit.

Commit Often

keeps your commits small and, again, helps you commit only related changes.

Rebase Often

keep your local working branch updated.

Don’t Commit Half-Done Work

You should only commit code when it’s completed.

If super big feature - split the feature’s implementation into logical chunks and remember to commit early and often.

Test Before You Commit

Test your code thoroughly to make sure it really is completed and has no side effects.

Write Good Commit Messages

Use the imperative, present tense ("change", not "changed" or "changes") to be consistent with generated messages from commands like git merge.

Use Branches

Neither use master nor develop when developing.

Always create new branch when starting new feature.

$ git branch RT-1234 origin/master
$ git branch feature/RT-1234 origin/master
$ git checkout -b feature/RT-1234 origin/master

Be Explicit

$ git push
$ git push

$ git push origin my-local-branch
$ git push --force
$ git push --force

vs.

$ git push origin my-local-branch --force
--force

vs.

--force-with-lease

Merging vs. Rebasing

$ git pull
vs.
$ git rebase
$ git fetch origin master
$ git merge origin/master
$ git fetch origin master
$ git rebase origin/master
$ git pull origin master --rebase

Rewriting History

$ git commit --amend
$ git rebase -i HEAD~4
pick 07c5abd add login page
pick de9b1eb patch login page
pick 3e7ee36 some typosssss
pick fa20af3 fix something

#
# Commands:
#  p, pick = use commit
#  r, reword = use commit, but edit the commit message
#  e, edit = use commit, but stop for amending
#  s, squash = use commit, but meld into previous commit
#  f, fixup = like "squash", but discard this commit's log message
#  x, exec = run command (the rest of the line) using shell
#
# These lines can be re-ordered; they are executed from top to bottom.

#
